# Delta Design

Enhancement paradigm for existing codebases. Instead of full architecture, map the existing structure and design only the delta — what changes, what stays, and what could break.

## When to Use

- Adding a feature or extending behavior in existing code
- Modifying existing workflows or data flows
- Any task that touches an existing codebase but isn't a simple bugfix or small addition

## Phase Sequence

1. [validate-requirements](../../phases/validate-requirements.md)
2. [impact-analysis](../../phases/impact-analysis.md)
3. [test-plan](../../phases/test-plan.md) *(scoped to the enhancement)*
4. [extract-contracts](../../phases/extract-contracts.md) *(optional — L/XL with >=2 independent affected modules)*
5. [implement](../../phases/implement.md) *(receives impact doc instead of arch doc)*
6. [code-review](../../phases/code-review.md)
7. [verify](../../phases/verify.md)
8. [deliver](../../phases/deliver.md)

## Phase Overrides

### test-plan
- Scope to the enhancement only — not the entire system
- Feature-level acceptance criteria (e.g., "when user does X, system does Y")
- NOT unit-test level — unit tests are generated by the agent during implementation

### extract-contracts
- Optional — only for L/XL enhancements with >=2 independent affected modules
- Input is the impact doc (not an architecture doc) — extract contracts from the affected modules section
- Skip for S/M complexity or enhancements touching a single module

### implement
- Receives the impact doc (not an architecture doc)
- Must preserve all existing behavior identified in the impact analysis
- Agent generates unit tests during implementation as needed
- If impact doc lists interface changes, implement them first
- L/XL with >=2 independent modules and extract-contracts completed: follow [Parallel Execution Protocol](../parallel-execution.md)

## Composition Rules

### + tdd
When composing with a TDD paradigm:
1. validate-requirements
2. impact-analysis
3. test-plan
4. extract-contracts *(optional — L/XL with >=2 independent modules)*
5. write-tests (from TDD paradigm, scoped to enhancement)
6. implement
7. code-review
8. verify
9. deliver
