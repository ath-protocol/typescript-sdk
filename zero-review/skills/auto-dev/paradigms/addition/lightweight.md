# Lightweight

Addition paradigm for single, well-scoped pieces of new functionality that fit cleanly into existing architecture without modifying it.

## When to Use

- Single well-scoped function, component, or endpoint
- Fits existing architecture unchanged
- Touches 1-2 modules at most
- No interface changes to existing code

## Phase Sequence

1. [validate-requirements](../../phases/validate-requirements.md) *(must confirm scope is truly small)*
2. [test-plan](../../phases/test-plan.md) *(lightweight, feature-level acceptance criteria)*
3. [implement](../../phases/implement.md) *(read surrounding code first to match patterns)*
4. [code-review](../../phases/code-review.md)
5. [verify](../../phases/verify.md) *(compiles, acceptance criteria pass, existing tests pass)*
6. [deliver](../../phases/deliver.md)

## Phase Overrides

### validate-requirements
- Must confirm the scope is truly small
- If 3+ modules are affected → escalate to `enhancement/delta-design`
- If interface changes are needed → escalate to `enhancement/delta-design`

### test-plan
- Kept lightweight — feature-level acceptance criteria only
- e.g., "when user clicks scan, software shows all sensitive files"
- NOT unit-test level — unit tests are generated by the agent during implementation

### implement
- Read surrounding code first to match existing patterns and conventions
- Agent generates unit tests on the fly during implementation
- No architecture doc or impact doc — the code itself is the reference

### verify
- Code compiles/builds successfully
- Acceptance criteria from test-plan pass
- All existing tests still pass

## Escalation

If at any point during implementation it becomes clear that:
- 3+ modules need changes → escalate to `enhancement/delta-design`
- Interface changes are needed → escalate to `enhancement/delta-design`
- The addition reveals a bug → file separately as `bugfix/hypothesis-driven`
