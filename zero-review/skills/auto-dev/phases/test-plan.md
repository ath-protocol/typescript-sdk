# Phase: Test Plan (TPDD)

## Purpose
Define boundaries, forbidden zones, and Must Have checkpoints before architecture. The TestPlan's Must Haves map to architecture, scope implementation sessions, and gate delivery.

## Process

1. **Invoke the test-plan skill:**
   ```
   /test-plan "description"
   /test-plan path/to/requirements.md
   ```
2. **Output location:** `{output_root}/designs/testplan_{YYYYMMDD}_{slug}.md`
3. **Check experience first:** Review your agent's memory or pattern files for prior patterns before generating

See `skills/test-plan/SKILL.md` for generation process, section definitions, and output format.

## TestPlan Sections

| Section | Purpose |
|---------|---------|
| **Capability** | What the submodule does, inputs/outputs, call interface |
| **Boundary** | Per-submodule limits — what it handles and what it excludes |
| **Forbidden Zone** | Hard redlines (>=3 per submodule) — what AI cannot do |
| **Validation Plan** | Must Have / Need Have / Should Have checkpoints |
| **Failure & Edge Cases** | Expected behavior for each failure mode |
| **Audit/Logs** | What must be logged, what must not |
| **Security/Token Constraints** | Rate limits, sandbox requirements, data isolation |
| **Integration Tests** | Cross-submodule data flows, contract matches, error propagation |
| **E2E User Flows** | Complete user goals that chain multiple capabilities — the highest verification level (subsumes smoke tests) |
| **E2E Coverage Matrix** | Capability-to-goal mapping that makes coverage gaps visible at a glance |
| **Environment Spec** | Runtime environment for E2E verification (user-provided or auto-provisioned) |

## Testing Hierarchy

The TestPlan defines three levels of testing. Each level gates the next — do not proceed to a higher level until the lower level passes.

### Level 1: Capability Tests

Single capability works in isolation. These are the per-submodule Must Haves in the Validation Plan.

- **Scope:** Per submodule — specific inputs produce specific outputs
- **Granularity:** Feature/acceptance level — observable user-facing behavior, not function internals
  - **Good:** "When user clicks scan, software lists all sensitive files in the directory"
  - **Good:** "POST /login with valid credentials returns 200 + token"
  - **Bad:** "Function `parseConfig()` returns correct object when given valid YAML"
- **Coverage rules:**
  - Every Must Have in the Validation Plan must have a test
  - Every Failure & Edge Case must have a test
  - 100% Must Have coverage required before proceeding to Level 2

Unit tests are generated by the agent during implementation as needed. The TestPlan focuses on *what the software should do* rather than *how individual functions behave*.

### Level 2: Integration Tests

Data passes correctly between modules. These catch bugs where every submodule passes its own tests but the product fails when assembled.

- **Scope:** Cross-submodule boundaries — A's actual output is B's valid input
- **Coverage rules:**
  - Every pair of communicating submodules has at least one contract test (A's output matches B's expected input)
  - Every pair has at least one error propagation test (B fails → A handles it correctly)
  - If submodule A sends N distinct data types to B, each type needs a test
  - 100% integration coverage required before proceeding to Level 3
- **Skip condition:** Skip if single submodule

### Level 3: E2E Tests

Complete user goals, start to finish, spanning multiple capabilities. This is the highest and most important verification level.

The user doesn't use one feature in isolation — they chain capabilities together to accomplish a goal. E2E tests this full journey. If E2E passes, the system is alive and working — no separate smoke test needed.

- **Scope:** Full user journeys from start to finish, crossing submodule boundaries
- **Structure: Goal → Flow → Paths**
  - **Goal:** What the user is trying to accomplish (maps to one or more capabilities)
  - **Flow:** The complete sequence of steps from start to finish
  - **Paths:** Happy path + error/edge paths for each flow. Error paths should test mid-journey failures, not just invalid initial input.

- **Coverage rules (critical):**
  1. **Goal completeness:** List all user-facing capabilities from the TestPlan. Every capability must appear in at least one E2E goal. If a capability isn't exercised by any E2E flow, that's a coverage gap — add a flow or justify the omission.
  2. **Path completeness:** Every E2E goal must have at least a happy path and an error path.
  3. **Step verification:** Each step in the flow must have a verification method — not just the final outcome. If step 3 of 5 produces wrong output, catch it at step 3.
  4. **No partial execution:** When a real environment is available, every E2E scenario must be executed. Cherry-picking is not allowed. If a scenario cannot run, document exactly why.
  5. **Coverage matrix required:** The TestPlan must include a coverage matrix (see below).

- **Examples (generic — adapt to any software type):**
  - CLI tool: Install → configure rules → scan directory → generate report → report contains expected findings
  - Service: Authenticate → submit job → poll status → retrieve result → result is correct
  - App: Launch → create content → save → reopen → content persists
  - Library: Import → configure → call API → handle result → handle error case

- **Skip condition:** Skip if single submodule with no user-facing flows

### E2E Coverage Matrix

Every TestPlan with E2E tests must include a coverage matrix mapping capabilities to E2E goals. This makes gaps visible at a glance.

```
| Capability        | E2E Goals     | Covered? |
|-------------------|---------------|----------|
| File scanning     | E2E-1, E2E-3  | ✓        |
| Report generation | E2E-1         | ✓        |
| Configuration     | E2E-2         | ✓        |
| Export            | (none)        | ✗ — GAP  |
```

**Rules:**
- Every user-facing capability from the Capability sections must appear as a row
- Any row with no E2E goal is a gap that must be resolved before the TestPlan passes its quality gate
- Internal/infrastructure capabilities (logging, caching) don't need rows — only capabilities the user directly interacts with

## Environment Spec

The Environment Spec captures how the agent accesses a real, running environment for E2E verification. This is what makes the difference between "test passes in theory" and "the user can actually use it."

**Three scenarios:**

1. **User provided an environment** at task start (e.g., `docker exec -i my-container`, `ssh dev@host`, local shell) → record it here as-is.
2. **User wants auto-setup** → the `provision-environment` phase will populate this section.
3. **No environment available** → note it. E2E sections fall back to local verification. The agent should ask the user during verify: "An environment would let me test this for real — want to provide one or have me set one up?"

**Fields:**
| Field | Required | Description |
|-------|----------|-------------|
| `type` | yes | `docker`, `ssh`, `local`, or freeform |
| `exec` | yes | Command template to run commands in the environment (e.g., `docker exec -i abc123`, `ssh user@host`) |
| `workdir` | no | Working directory inside the environment (default: project root) |
| `ports` | no | Ports the app listens on |
| `env_vars` | no | Environment variables needed |
| `auto_provisioned` | no | `true` if set up by provision-environment phase |

**When an environment IS available:** The E2E User Flows must be written as **real user-facing tests** — test what the user actually experiences. These are complete user goals executed for real in the environment, with verification at every step. Not function-level checks, not API-level checks — the user's perspective, start to finish.

## Quality Gate
- Every Must Have is observable: specific input -> specific output
- Every Forbidden Zone has >=3 concrete redlines
- Submodule boundaries do not overlap
- Integration Must Haves cover every cross-submodule data flow (skip if single submodule)
- Every major user goal has at least one E2E flow with happy + error paths
- Every E2E scenario has a verification method at each step, executable by an AI agent
- E2E coverage matrix has no gaps (every user-facing capability covered by at least one E2E goal)
- If environment is available, E2E verification methods target the real environment

## Skip Conditions
This phase is never skipped in paradigms that include it. Not used in `bugfix/hypothesis-driven` (diagnosis doc defines correctness criteria instead).
