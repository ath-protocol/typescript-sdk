# Contract: Artifact

Describes a built, runnable artifact that auto-test can provision and exercise.

**Producer:** `auto-dev` (or the project's existing build system)
**Consumer:** `auto-test`

## Schema

```yaml
project_type: web | cli | api | desktop | library
name: string                          # Project or artifact name

build:
  method: dockerfile | compose | script | prebuilt
  dockerfile: string                  # Path, if method is dockerfile
  compose_file: string                # Path, if method is compose
  build_command: string               # Shell command, if method is script
  image: string                       # Image reference, if method is prebuilt

run:
  command: string                     # How to start the artifact
  port: number                        # Primary port, if network-accessible
  additional_ports: [number]          # Secondary ports (admin, debug, etc.)
  environment_variables:              # Required env vars
    - name: string
      value: string                   # Can be placeholder: "${DATABASE_URL}"
      required: boolean

healthcheck:
  method: http | tcp | command
  target: string                      # URL, host:port, or shell command
  expected: string                    # HTTP 200, exit 0, specific output
  timeout_seconds: number
  retries: number

entrypoints:                          # How a user would start interacting
  - type: url | command | import
    value: string                     # http://localhost:3000, "mytool --help", "import mylib"
    description: string               # "Main web interface", "CLI help", "Python package"
```

## Rules

- At minimum, `project_type`, `run.command`, and one `entrypoint` must be populated.
- `healthcheck` is strongly recommended. Without it, auto-test must guess when the artifact is ready, which causes flaky sessions.
- `environment_variables` with `required: true` must be resolved before auto-test can provision. If values are unknown, escalate to sponsor.
- `build.method: prebuilt` means the image already exists (e.g., published to a registry). No build step needed.
- This contract can be generated by auto-dev's deliver phase, extracted from the project's existing configuration (docker-compose.yml, package.json), or written manually.
